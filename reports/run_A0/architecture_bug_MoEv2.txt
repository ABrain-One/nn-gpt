=== ARCHITECTURE BUG: MoEv2 — Hardcoded Spatial Dimension ===

Error (126x across all datasets):
  RuntimeError: mat1 and mat2 shapes cannot be multiplied (64x131072 and 2048x128)

Affected model : MoEv2
Affected runs  : all 18 variants (3 losses × 6 optimizers) × 7 datasets = 126

Root cause
----------
ConvFeatureExtractor in MoEv2 hardcodes its flattened output dimension:

    # base_nn/MoEv2.py, line 89
    self.output_dim = 128 * 4 * 4   # assumes 32×32 input (CIFAR-10)

The model applies 3× MaxPool2d(2,2), so spatial dims shrink by ÷8.
For a 32×32 input:  32 → 16 → 8 → 4  →  128*4*4 = 2048  ✓
For a 256×256 input: 256 → 128 → 64 → 32  →  128*32*32 = 131072  ✗

Gate and Expert linear layers are built with input_dim=2048.
Any dataset whose images are larger than 32×32 causes a matmul failure.

Datasets affected vs not
-------------------------
  Fail (image > 32×32): imagenette (224px), celeba (218px), places365 (256px),
                         mnist (28px → resized to 256px by eval), svhn (32px*)
  Would pass (32×32)  : cifar-10, cifar-100
  (* svhn is 32×32 but was in the failing underscore-named batch)

Fix (requires changing base_nn — currently treated as ground truth)
--------------------------------------------------------------------
Replace the hardcoded pool-and-flatten in ConvFeatureExtractor with
AdaptiveAvgPool2d so output_dim is always 128, regardless of input resolution:

    # Instead of:
    self.pool = nn.MaxPool2d(2, 2)
    self.output_dim = 128 * 4 * 4

    # Use:
    self.adaptive_pool = nn.AdaptiveAvgPool2d(1)
    self.output_dim = 128
